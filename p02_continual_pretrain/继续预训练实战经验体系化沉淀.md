## 第 1 章 概念与定位

### 1.1 四个训练阶段的区分

```
Pretrain (从零预训练)
    │  数据：万亿 token 通用语料；lr：1e-4~6e-4；目标：学语言基础
    ▼
Continual Pretrain (继续预训练) ← 本文聚焦
    │  数据：1%~10% 原数据量的领域/语言语料；lr：1e-6~1e-5
    │  目标：注入新知识，不破坏原能力
    ▼
SFT (监督微调)
    │  数据：万~百万条指令数据；lr：5e-6~2e-5
    │  目标：学会对话格式、指令遵循
    ▼
RLHF / DPO
       数据：偏好对；目标：对齐人类偏好
```

### 1.2 继续预训练的三种典型场景

| 场景 | 目标 | 数据来源 | 典型数据量（7B 模型） |
|---|---|---|---|
| **语言适配** | 英文模型学中文 | Wiki / 新闻 / 书籍 | 10B ~ 50B tokens |
| **领域注入** | 通用模型学法律/医疗/代码 | 行业语料 | 1B ~ 20B tokens |
| **时间更新** | 知识截止日期向后推 | 最新新闻 / 论文 | 1B ~ 10B tokens |

### 1.3 为什么"继续预训练"比"从零训练"难

从零训练是**在空白权重上塑形**，只要数据足够、lr 正确，结果就会出现；
继续预训练是**在一个已经精细调好的权重分布上做微扰**，稍有不慎就会：

- **拉偏分布**（模型只会说新领域的话）
- **破坏对话能力**（embedding 被改坏）
- **灾难遗忘**（忘掉英文、代码、数学）

这也是为什么业界训一个 base 模型容易，训一个"既懂医疗又能聊天"的 continual 模型反而很难。

## 第 2 章 数据：继续预训练的地基

### 2.1 数据量的三档参考（按模型规模）

**核心经验法则：继续预训练数据量 ≈ 原预训练量的 1% ~ 10%**

| 模型规模 | 领域适配 | 知识注入 | 能力增强（代码/数学）|
|---|---|---|---|
| **0.5B~1B** | 100M ~ 1B tokens | 50M ~ 500M | 500M ~ 2B |
| **3B~7B** | **1B ~ 10B tokens** | 500M ~ 5B | 5B ~ 50B |
| **13B** | 5B ~ 30B | 2B ~ 10B | 20B ~ 100B |
| **70B** | 20B ~ 100B | 10B ~ 50B | 50B ~ 500B |

> **占比判断**：
> - < 0.1%：几乎无感知，只能证明 pipeline 通（如本实验的 34M/12T ≈ 0.0003%）
> - 1% ~ 5%：**甜蜜点**，能学到新东西且遗忘可控
> - \> 10%：需要混合通用数据对冲，否则必然大面积遗忘

### 2.2 数据质量比数量更重要

一条**质量判断的实操清单**：

- [ ] **去重**：MinHash + LSH 做文档级去重（重复数据会让模型过拟合字面）
- [ ] **去噪**：HTML 残留、乱码、广告、版权声明要清
- [ ] **长度分布**：太短（< 50 tokens）的丢掉，packing 效率低
- [ ] **perplexity 过滤**：用 base 模型算 ppl，砍掉**最高 10%**（乱码）和**最低 5%**（重复模板）
- [ ] **领域混合比例**：严格按目标配比，不要拍脑袋
- [ ] **语言检测**：用 `fasttext-langdetect`，非目标语言的 < 5%

### 2.3 Packing 还是 Padding？

| 模式 | 训练效率 | 语义完整 | 实现复杂度 | 推荐场景 |
|---|---|---|---|---|
| **Padding** | 低（短文本浪费 60%+ 算力）| ✅ 每条独立 | 简单 | 文档平均长度接近 max_seq_length 时 |
| **Packing** | 高（利用率 99%+）| ⚠️ 跨文档边界 | 中等 | **继续预训练默认选 Packing** |
| **Packing + document mask** | 高 | ✅ 无泄漏 | 高 | 严格生产环境 |

**细节**：Packing 时**务必在文档间插入 `eos_token`** 作为分隔。本项目 `dataset.py` 的实现就是这样（见 `PackedDataset._pack`）。

### 2.4 混合通用数据防遗忘（Replay）

| 配比（新领域 : 通用）| 适用 | 风险 |
|---|---|---|
| 1 : 3（保守）| 轻度领域适配 | 学不透新领域 |
| **1 : 1（平衡，推荐）** | 大多数场景 | — |
| 3 : 1（激进）| 数据量充足的语言适配 | 必须做 eval 监控 |
| 10 : 0（纯新领域）| 通用能力不重要的场景 | 几乎必定灾难遗忘 |

**通用数据从哪来**：
1. **开源复刻**：The Pile、RedPajama 的中文部分、SlimPajama
2. **模型自蒸馏**：用 base 模型自己生成文本（防止分布漂移最稳）
3. **保留原语料采样**：如果知道原模型用什么数据训的最好（难）

## 第 3 章 训练超参数的"刻度"

### 3.1 学习率——最关键的一个旋钮

继续预训练的 lr 选择本质是在问：**"我想让参数动多大"**。

| 场景 | 推荐 lr 区间 | 备注 |
|---|---|---|
| 全参 · 小模型 (< 1B) | **1e-6 ~ 3e-6** | 本实验用 5e-6 稍大，导致 embedding 变化 4.27% |
| 全参 · 中模型 (7B~13B) | 5e-6 ~ 2e-5 | Llama/Qwen 继续预训练常用区间 |
| 全参 · 大模型 (70B+) | 2e-5 ~ 5e-5 | 参数多冗余大，可以更大 |
| **LoRA 全尺寸** | **5e-5 ~ 3e-4** | LoRA 只调适配器，lr 要比全参高一两个数量级 |
| QLoRA | 1e-4 ~ 5e-4 | 和 LoRA 接近，但要更仔细调 warmup |

**经验公式**：
```
继续预训练 lr ≈ 原预训练 lr × 1/10 ~ 1/100
```

### 3.2 Batch Size 与 Gradient Accumulation

**真正起作用的是 global batch size = per_device × GA × world_size**

| 模型规模 | 推荐 global batch (tokens) |
|---|---|
| 0.5B | 128K ~ 512K tokens |
| 7B | 1M ~ 4M tokens |
| 70B | 4M ~ 16M tokens |

换算：global_batch_tokens = `seq_len × per_device_bs × GA × num_gpus`

本实验：`512 × 4 × 4 × 1 = 8192` tokens/step，**对 0.5B 偏小**，造成 loss 曲线抖动较大。

### 3.3 Scheduler 选择

| Scheduler | 曲线形状 | 继续预训练场景 |
|---|---|---|
| **cosine**（本实验用）| 先 warmup，后 cos 衰减到 0 | ✅ **默认选择**，业界用得最多 |
| linear | 线性衰减到 0 | SFT 常用 |
| constant with warmup | warmup 后恒定 | 长期训练（>100B tokens）用 |
| cosine_with_min_lr | 最低 lr 不为 0 | **大规模多阶段训练推荐**，防止末期 lr=0 浪费算力 |

**Warmup ratio**：小数据继续预训练 **3%~5%** 足够（本实验 5%），长期训练用 1%。

### 3.4 其他关键参数

```python
weight_decay = 0.01          # 继续预训练保持 0.01；全参时 >0.05 会拖累 embedding
max_grad_norm = 1.0          # 梯度裁剪，bf16 下这个很重要
warmup_ratio = 0.03 ~ 0.05   # 数据越多越小
bf16 = True                  # 现代 GPU 默认 bf16，不要用 fp16（易 NaN）
gradient_checkpointing = True # 24G 以下必开
```

### 3.5 训练轮数（epoch）

**继续预训练几乎永远只跑 1 epoch**。原因：

1. 数据去重后，**见第二次就是过拟合字面**
2. 大模型本身样本效率极高
3. 想见更多数据 → 加数据，不是加 epoch

**例外**：数据量远小于模型容量（如 10M tokens 训 7B），可以跑 2~3 epoch，但要监控 eval loss。

## 第 4 章 Loss 的读法与判异

### 4.1 Loss 到底是什么（必须搞清楚）

**CLM 训练的 loss = 一个 batch 中所有非 padding token 的平均交叉熵**：

$$
L = -\frac{1}{N}\sum_{i} \log P(t_{i+1} \mid t_1, ..., t_i)
$$

HF Trainer 打印的 `loss: 3.12` 是 `logging_steps` 内的滑动平均（默认 10 步）。

**Perplexity = e^loss**，是更直观的"有效候选数"：

| loss | PPL | 含义（Qwen 词表 15 万）|
|---|---|---|
| 11.9 | 151936 | 完全随机猜 |
| 5.0 | 148 | 刚学会基本分布 |
| 3.0 | 20 | **通用中文 base 水平** |
| 2.0 | 7.4 | 强 base 或小领域过拟合边缘 |
| 1.0 | 2.7 | 强记忆或接近 ground truth |
| 0.5 | 1.6 | **大概率数据泄漏** |

### 4.2 健康 Loss 曲线的特征

```
loss
 │
 │╲          ← warmup 期：lr 小，loss 快速下降 1~2 个点
 │ ╲╲
 │   ╲╲___      ← 稳定下降期：每 1000 步降 0.1~0.2
 │       ╲_____
 │            ╲____  ← 末期收敛
 └────────────────→ step
```

### 4.3 异常 Loss 曲线的诊断

| 现象 | 可能原因 | 排查方向 |
|---|---|---|
| Loss 起点 > 8 | tokenizer 不匹配 / 模型未加载预训练权重 | 检查 `from_pretrained` 路径 |
| Loss 起点 < 2 | 数据在预训练集里泄漏 | 换评估集验证 |
| Loss 直接 NaN | fp16 溢出 / lr 太大 | 换 bf16，降 lr |
| Loss 抖动 >1.0 | batch size 太小 / 数据方差大 | 加 GA 或增大 GB |
| Loss 不下降 | lr 太小 / 被冻结了 | 打印 `model.parameters().requires_grad` |
| Loss 突然跳崖 | 数据有 bug / 特殊 token 异常 | dump 该 step 的 batch 看 |
| 验证 loss 上升 | 过拟合或灾难遗忘 | 减 epoch 或加 replay |

### 4.4 本实验的 loss 读解

```
起点 3.12 → 500 步 3.03（LoRA）/ 4200 步 ~2.85（全参）
```
- 起点 3.12 符合 Qwen2.5-0.5B 在中文 wiki 上的预期
- 下降 0.2 ~ 0.3 属于**温和注入**，不会学出"只会写 wiki 体"的问题
- 全参 loss 降得比 LoRA 多 0.15，但代价是 embedding 变化 4.27%，**得不偿失**

## 第 5 章 全参 vs LoRA：选型决策

### 5.1 本实验的实测对比（同数据、同 tokenizer、H20 单卡）

|  | **全参训练** | **LoRA 训练** |
|---|---:|---:|
| 训练步数 | 4220 | 500 |
| 可训参数 | 494M (100%) | 35M (6.65%) |
| 显存占用 | ~38 GB | ~18 GB |
| 训练时长 | ~55 min | ~5 min |
| **全局权重变化** | **1.03%** | **0.89%** |
| **embedding 层变化** | **🔴 4.27%** | **🟢 0.00%** |
| norm 层变化 | 0.0025% | 0.00% |
| 保存大小 | 1.0 GB | **66 MB**（仅适配器）|

**结论**：LoRA 用 **1/10 的时间、1/2 的显存、1/15 的存储**，得到了相同量级的学习效果，**且完全不碰 embedding**。

### 5.2 决策树

```
                 开始继续预训练
                      │
          ┌───────────┴───────────┐
          │                       │
   数据 > 1B tokens?          数据 < 1B tokens?
          │                       │
     ┌────┴────┐             ┌────┴────┐
     │         │             │         │
  是领域     是语言        是领域    是语言
  增强？    适配？         注入？   适配？
     │         │             │         │
   全参     全参+冻结     LoRA    LoRA（大 rank）
           embedding       r=64     r=128~256
```

### 5.3 LoRA 的关键超参

```python
LoraConfig(
    r = 64,                     # 继续预训练推荐 64~128（比 SFT 的 8~32 大）
    lora_alpha = 128,           # 惯例 alpha = 2 × r
    lora_dropout = 0.05,        # 继续预训练数据大可以 0；小数据 0.05~0.1
    target_modules = [
        "q_proj", "k_proj", "v_proj", "o_proj",   # 注意力 4 个都加
        "gate_proj", "up_proj", "down_proj",       # MLP 3 个都加
    ],
    # 继续预训练不建议加 embedding 和 lm_head（会破坏 tied weight）
    bias = "none",
)
```

**为什么 target_modules 要全加？**
- 只加 `q,v`（SFT 常见做法）：适合改"指令风格"
- **加全 7 个**：适合学"新知识"（继续预训练场景）

### 5.4 LoRA 的 "等效 lr" 直觉

LoRA 更新量 ≈ `B @ A × (alpha / r) × lr`，所以：
- alpha=128, r=64 时，缩放因子是 2
- LoRA lr = 1e-4 ≈ 等效全参 lr = 2e-4 × 局部秩限制

这解释了为什么 LoRA lr 要设得比全参大 10~100 倍。

## 第 6 章 灾难性遗忘：真正的头号敌人

### 6.1 什么是灾难性遗忘

模型在新数据上学得越多，原来的能力丢得越多。**这是继续预训练 90% 坑的根源**。

### 6.2 本实验的典型表现

> 用户反馈：跑完全参训练后，原本能"续写"百科的模型，**基本 prompt 输出都开始混乱**，选择题能力也崩了。

诊断如下：

| 症状 | 根因 | 证据 |
|---|---|---|
| 对话模板失效 | embedding 被拉偏 | **embedding 变化 4.27%**（v.s. LoRA 的 0%）|
| 输出风格僵硬百科体 | 所有 MLP 向 wiki 分布靠 | MLP 层变化 1.19% |
| 英文/代码能力下降 | 底层 attention 被改 | 前几层 v_proj 变化 2% |
| loss 下降但 ppl 评测变差 | 过拟合 wiki 分布 | 需独立 ppl 评测 |

### 6.3 防遗忘的五大武器

#### 武器 1：LoRA（最省事）
适配器是**加法项**，原权重物理冻结，遗忘从机制上就不可能。

#### 武器 2：冻结关键模块
```python
for name, p in model.named_parameters():
    if "embed_tokens" in name or "lm_head" in name:
        p.requires_grad = False
    if "norm" in name:
        p.requires_grad = False
```
只训练 attention 和 mlp，保住 embedding 的"字典"。

#### 武器 3：Replay Buffer（经典）
把原通用数据按 1:1 ~ 1:3 混进新数据。**工业界做多语言继续预训练的标准做法**。

#### 武器 4：学习率保护
```
lr_new_domain = 5e-6
lr_embedding = 5e-7   # 给 embedding 单独小 10 倍的 lr
lr_norm = 0           # norm 直接不训
```
通过参数组实现（见第 7.3 节）。

#### 武器 5：EMA / SWA
训练中保留一个指数移动平均的权重副本，输出时混合：
```
W_final = 0.5 × W_base + 0.5 × W_trained
```
简单粗暴，防遗忘效果好，但会稀释新知识。

### 6.4 衡量遗忘的量化方法

**核心指标：权重相对变化率**
$$
\Delta_{rel} = \frac{\|W_{trained} - W_{base}\|_F}{\|W_{base}\|_F}
$$

**判断区间**（本文实测给出的经验）：

| 全局 Δ | embedding Δ | 健康度 | 后果 |
|---|---|---|---|
| < 0.3% | < 0.1% | 几乎没训 | 浪费算力 |
| 0.3% ~ 2% | < 0.5% | 🟢 理想 | 新知识注入 + 原能力保留 |
| 2% ~ 5% | 0.5% ~ 2% | 🟡 警戒 | 对话开始飘 |
| \> 5% | \> 2% | 🔴 危险 | 灾难遗忘 |

**本实验数据**：
- 全参：全局 1.03% / embedding 4.27% → 看似全局健康，但 embedding 进入红区
- LoRA：全局 0.89% / embedding 0.00% → 完全健康

**这也揭示了一个重要事实：看全局变化率会骗人，必须看分层。**

### 6.5 防遗忘的"最小代价方案"

如果你就是要用全参训练，最低成本的防遗忘配方：

```python
# 1. lr 再降一档
learning_rate = 1e-6          # 而不是 5e-6

# 2. 冻结 embedding 和 lm_head
for n, p in model.named_parameters():
    if "embed_tokens" in n or "lm_head" in n:
        p.requires_grad = False

# 3. 加 replay
# 数据集 = 新领域数据 + base 模型自蒸馏文本，比例 1:1

# 4. 短训
num_train_epochs = 1
max_steps = 2000      # 不要跑到底
```

## 第 7 章 工程实战：显存、速度、稳定性

### 7.1 显存占用公式（bf16 训练）

$$
\text{显存} \approx \underbrace{2N}_{权重} + \underbrace{2N}_{梯度} + \underbrace{8N}_{Adam 优化器} + \underbrace{B \cdot L \cdot H \cdot k}_{激活}
$$

- N：参数量（字节级），bf16 下 1 param = 2 bytes
- 总计 **每 1B 参数 ≈ 12 GB**（权重+梯度+优化器）
- 激活按 `batch × seq × hidden × 层数` 计算，通常 5~20 GB

**示例（Qwen2.5-0.5B 全参）**：
- 权重+梯度+优化器 ≈ 6 GB
- 激活（bs=4, seq=512, GC 开）≈ 3 GB
- 实测 ~15 GB，预留 buffer 到 20 GB

### 7.2 显存优化层级（从低痛苦到高痛苦）

1. **bf16 混合精度** - 免费拿到 2x 显存空间
2. **Gradient Checkpointing** - 30% 显存省，20% 速度降
3. **LoRA/QLoRA** - 显存省 50%+，几乎无速度损失
4. **DeepSpeed ZeRO-2** - 优化器分片，单卡省不到，多卡线性省
5. **DeepSpeed ZeRO-3** - 权重也分片，极致省显存但慢 2~3 倍
6. **CPU offload** - 把优化器放 CPU 内存，极慢但能训大模型
7. **FlashAttention-2** - 显存省 + 速度加（如有条件必开）

### 7.3 参数分组（手动控制谁动多少）

```python
# 经典分组：embedding 慢学、norm 不学
optimizer_grouped_parameters = [
    {
        "params": [p for n, p in model.named_parameters()
                   if "embed" in n or "lm_head" in n],
        "lr": 5e-7,              # embedding 10 倍慢
        "weight_decay": 0.0,     # embedding 不加 weight decay
    },
    {
        "params": [p for n, p in model.named_parameters()
                   if "norm" in n or "bias" in n],
        "lr": 0.0,               # norm/bias 冻结
        "weight_decay": 0.0,
    },
    {
        "params": [p for n, p in model.named_parameters()
                   if not any(k in n for k in ["embed", "lm_head", "norm", "bias"])],
        "lr": 5e-6,              # 其他正常训
        "weight_decay": 0.01,
    },
]
```

### 7.4 常见工程坑（来自本实验踩的坑）

| 坑 | 表现 | 解决 |
|---|---|---|
| FusedAdam 编译失败 | "GCC 版本太低 / missing Python.h" | 设 `DS_BUILD_FUSED_ADAM=0`，或显式传入 `torch.optim.AdamW` |
| LoRA + GC 报 "grad not required" | backward 报错 | 加 `model.enable_input_require_grads()` |
| DeepSpeed import 失败 | accelerate 硬依赖 deepspeed | 临时重命名 deepspeed 目录，或升级 accelerate |
| tokenizer pad_token 错误 | "PAD token not found" | `tokenizer.pad_token = tokenizer.eos_token` |
| gradient_accumulation 配置冲突 | "HF=1 vs DS=4" | DS config 里设 `"auto"` |
| `use_cache=True` 和 GC 冲突 | 警告（不影响）| 训练前 `model.config.use_cache=False` |

## 第 8 章 评测方法论

继续预训练**最容易被忽视的一步**：没有评测，你根本不知道自己是在进步还是在退步。

### 8.1 四层评测体系

```
┌───────────────────────────────────────┐
│ L4. 下游任务（SFT 后再评）             │  — C-Eval、MMLU、HumanEval
├───────────────────────────────────────┤
│ L3. Zero-shot 生成质量（人工打分）     │  — 给一批 prompt 让模型续写
├───────────────────────────────────────┤
│ L2. Loglikelihood 选择题评测           │  — MMLU 的 lm-eval 方式
├───────────────────────────────────────┤
│ L1. Perplexity（必做！）                │  — 训练 / 验证 / 通用 / 领域 四份
└───────────────────────────────────────┘
```

### 8.2 Perplexity 评测（最重要）

**准备四份独立的评测集**，都不在训练集里：

```python
eval_sets = {
    "train_domain":   "抽 1000 条训练数据（看是否在学）",
    "val_domain":     "同领域但模型没见过（看是否泛化）",
    "general":        "通用中文文本（看是否遗忘中文）",
    "english":        "英文文本（看是否遗忘英文）",
}

# 每个 checkpoint 都评一次，画曲线：
# - train_domain ppl 下降 = 在学
# - val_domain ppl 下降 = 泛化好
# - general ppl 不升 = 没遗忘
# - english ppl 不升 = 没遗忘英文
```

**本实验应该做但没做的**：建立通用中文/英文的 eval 集，对比全参和 LoRA 在 general ppl 上的差距。

### 8.3 Base 模型评测的正确姿势

**Base 模型不能用"聊天 prompt"评！** 这是新人最容易犯的错。

| 错误方式 | 正确方式 |
|---|---|
| "请问中国首都是哪里？" | "中国的首都是" |
| "A/B/C/D 选哪个？" | 用 **loglikelihood**：算 `P(Prompt+A)`、`P(Prompt+B)`... 比大小 |
| 人工看回答 | 批量 ppl 对比 |

**选择题 loglikelihood 评测代码骨架**：
```python
def eval_choice(model, tokenizer, prompt, choices):
    scores = []
    for c in choices:
        full = prompt + c
        ids = tokenizer(full, return_tensors="pt").input_ids.to(model.device)
        with torch.no_grad():
            out = model(ids, labels=ids)
        # 取 choice 部分的 loss（注意 mask 掉 prompt）
        scores.append(-out.loss.item())
    return choices[scores.index(max(scores))]
```

这正是 MMLU、C-Eval 的标准做法。

### 8.4 权重变化量分析（我们的独门武器）

见附录 A 的 `diff_weights.py`。关键是**看分层而不是全局**。

## 第 10 章 常见误区与 FAQ

### Q1：继续预训练后，模型对话能力反而变差，是 bug 吗？
**不是 bug，是灾难遗忘**。特别是全参训练小模型时。解决方案见第 6 章。

### Q2：Loss 降得越多越好吗？
**不是**。训练集 loss 可以降到 0.5，但那时模型已经过拟合，general ppl 会爆炸。健康下降是 0.2~0.5。

### Q3：我数据只有几十 MB，能做继续预训练吗？
**能跑通，但不建议做全参**。几十 MB 数据对 0.5B 模型是 0.01% 的增量，用 LoRA 更合适。全参必然灾难遗忘。

### Q4：为什么我的 base 模型不会回答问题？
**Base 模型本来就不会回答，它只会续写**。回答问题需要 SFT。对 Base 模型的正确 prompt 是 "中国的首都是"，而不是 "中国的首都是哪里？"。

### Q5：LoRA 继续预训练学不到新知识吗？
**能学，但能力上限低于全参**。LoRA 擅长学"组合用法"，不擅长学"新事实"。如果目标是注入 10 万条事实，需要大 rank（128+）或改用全参。

### Q6：学习率怎么调才对？
**四步法**：
1. 从 `1e-5` 起跳（中等模型）或 `1e-6`（小模型）
2. 跑 100 步看 loss：不降就 2x，爆炸就 /10
3. 稳定后跑 500 步，grad_norm 稳定在 1~10 就对了
4. 正式训练时再 /2（比试跑的保守）

### Q7：什么时候需要跑完整训练？什么时候可以早停？
**没有验证集 loss 提升时**可以早停。更实际的做法：每 1000 步评一次 general ppl，升了就停。

### Q8：继续预训练完的模型，能直接上线用吗？
**几乎不能**。Base 模型不会按指令对话，**必须接 SFT**。完整路径：Base → Continual Pretrain → SFT → DPO。

### Q9：如何选 LoRA 的 rank？
| rank | 场景 |
|---|---|
| 8~16 | SFT 指令微调 |
| 32~64 | 领域 SFT / 轻量继续预训练 |
| 64~128 | **本文场景：继续预训练** |
| 128~256 | 数据量很大的继续预训练 |

### Q10：训练过程 NaN 了怎么办？
- 第一步：换 bf16（不要用 fp16）
- 第二步：lr /10
- 第三步：检查数据有没有超长行或特殊字符
- 第四步：加 `max_grad_norm=0.5`
- 最后：看 checkpoint 在 NaN 前是什么数据

## 附录 A：权重变化量分析工具

代码见同目录 [`diff_weights.py`](./diff_weights.py)。核心逻辑：

```python
for name, p_base in base_params.items():
    p_trained = trained_params[name]
    diff = (p_base.float() - p_trained.float())
    rel_change = diff.norm() / p_base.norm() * 100

    # 分组统计：embedding / attention / mlp / norm / lm_head
    group[classify(name)]["diff"] += diff.norm() ** 2
    group[classify(name)]["base"] += p_base.norm() ** 2
```

**输出四份报告**：
1. 全局相对变化率
2. 按模块分组的变化率（**最重要，看 embedding！**）
3. Top-10 变化最大的参数
4. 健康度自动判定
