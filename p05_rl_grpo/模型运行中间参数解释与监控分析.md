## 1. 监控方法与工具总览

### 1.1 monitor.py 核心功能

`monitor.py` 提供了以下四大监控能力：

| 功能模块 | 函数 | 作用 |
|---------|------|------|
| 日志加载 | `load_training_logs()` | 从 `trainer_state.json` 中读取完整训练历史 |
| 曲线绘制 | `plot_reward_curves()` | 绘制奖励/KL/Loss 曲线，支持多组实验对比 |
| 奖励欺骗检测 | `detect_reward_hacking()` | 通过三重检验判断是否发生 Reward Hacking |
| 生成质量抽查 | `sample_quality_check()` | 实际推理验证准确率，对比训练奖励 |

### 1.2 使用方式

```bash
# 默认监控：绘制奖励/KL/Loss 曲线
python monitor.py --log-dir outputs/grpo

# 多组消融实验对比
python monitor.py --compare outputs/grpo_g4 outputs/grpo_g8 outputs/grpo_g16

# 奖励欺骗检测
python monitor.py --check-hacking outputs/grpo

# 生成质量抽查
python monitor.py --quality-check outputs/grpo/final
```

### 1.3 观测经验总结

| 观测维度 | 核心指标 | 观测频率 | 判断方式 |
|---------|---------|---------|---------|
| 训练收敛 | loss, reward | 每 5-10 步 | loss 稳定下降，reward 稳步上升 |
| 策略稳定 | kl, clip_ratio | 每 5-10 步 | KL 缓慢增长但不发散 |
| 生成质量 | completions/* | 每 10-20 步 | 生成长度适中、方差合理 |
| 风险预警 | reward_std, frac_reward_zero_std | 每 50-100 步 | 奖励方差不应过低或突降 |

### 2.1 损失与优化相关

#### `loss` = 0.1054

| 属性 | 说明 |
|------|------|
| **含义** | 策略梯度损失（Policy Gradient Loss），即 GRPO 目标函数的负值 |
| **计算方式** | $L = -\mathbb{E}\left[\min\left(\frac{\pi_\theta}{\pi_{\text{old}}} \hat{A}, \text{clip}\left(\frac{\pi_\theta}{\pi_{\text{old}}}, 1-\epsilon, 1+\epsilon\right) \hat{A}\right)\right] + \beta \cdot D_{KL}(\pi_\theta \| \pi_{\text{ref}})$ |
| **健康范围** | 0.01 ~ 1.0（取决于训练阶段和奖励量级） |
| **观测要点** | 应逐步下降并趋于稳定；剧烈震荡说明学习率过大或奖励信号不稳定 |

**异常诊断：**
- loss 持续不降 → 学习率过低，或奖励函数提供不了有效梯度信号
- loss 发散（急剧上升） → 学习率过大，梯度爆炸
- loss 降到接近 0 后又回升 → 可能发生了策略崩塌（policy collapse）

---

#### `grad_norm` = 5.65625

| 属性 | 说明 |
|------|------|
| **含义** | 梯度的 L2 范数，反映当前梯度的大小 |
| **计算方式** | $\|\nabla_\theta L\|_2 = \sqrt{\sum_i g_i^2}$ |
| **健康范围** | 0.1 ~ 10.0（在 `max_grad_norm=1.0` 裁剪后通常 ≤ 1.0，此处值高于 1 说明裁剪前的原始梯度值） |
| **观测要点** | 应保持相对稳定，偶尔波动正常 |

**异常诊断：**
- grad_norm 持续 > 50 → 梯度爆炸，需降低学习率或增强梯度裁剪
- grad_norm 接近 0 → 梯度消失，模型可能停止学习
- grad_norm 突然跳变 → 遇到了异常样本或奖励信号突变

**与 max_grad_norm 的关系：**
```
实际更新梯度 = min(grad_norm, max_grad_norm) / grad_norm × 原始梯度
```
当 `grad_norm > max_grad_norm` 时，梯度被缩放以防止更新过大。

---

#### `learning_rate` = 1.63e-07

| 属性 | 说明 |
|------|------|
| **含义** | 当前步的学习率 |
| **调度策略** | 使用 cosine 调度，含 3% warmup |
| **典型范围** | RL 训练通常 1e-7 ~ 5e-6（远小于 SFT 的 1e-5 ~ 5e-5） |
| **观测要点** | 确认 warmup 阶段正常上升，后续按预期衰减 |

**为什么 RL 训练学习率远小于 SFT？**
- RL 的梯度方差更大（采样导致），大学习率容易震荡
- 策略变化过快会导致 KL 发散
- 小学习率保证策略的平稳更新

---

| **含义** | 生成文本被**长度截断**的比例（触及 max_completion_length 上限） |
| **健康范围** | < 0.1（低于 10%） |
| **观测要点** | 过高说明 max_completion_length 设置不够，模型回答被强制截断 |

**异常诊断：**
- clipped_ratio > 0.3 → 需增大 `max_completion_length` 或检查模型是否在生成无意义重复
- clipped_ratio = 0 → 说明生成长度远未触及上限，可适当缩小以节省算力

| **观测要点** | 与 mean_length 差距越大，说明被截断越多 |

#### `completions/min_terminated_length` = 107.0

#### `completions/max_terminated_length` = 430.8

| **含义** | 正常终止的回复中最长的长度 |
| **观测要点** | 如果远小于 max_length，说明大部分回复都能自然结束 |

### 2.3 奖励相关

#### `rewards/reward_fn/mean` = -0.1248

| 属性 | 说明 |
|------|------|
| **含义** | 奖励函数返回的**原始平均奖励值**（未经归一化） |
| **计算** | 对当前 batch 所有 (prompt, completion) 对调用 reward_fn 的平均值 |
| **健康范围** | 取决于奖励函数设计（本例中 correctness: -1.0 ~ +1.0） |
| **观测要点** | 应随训练逐步上升 |

---

#### `rewards/reward_fn/std` = 0.0661

| 属性 | 说明 |
|------|------|
| **含义** | 奖励函数原始分数的**标准差** |
| **观测要点** | 反映组内奖励的分散程度，这是 GRPO 能否学习的关键！ |

**重要理解：** GRPO 依赖组内的**相对排名**来计算优势。如果所有回复奖励几乎一样（std→0），优势信号消失，模型无法学习。

---

#### `reward` = -0.1248

| 属性 | 说明 |
|------|------|
| **含义** | 与 `rewards/reward_fn/mean` 相同，是汇总后的平均奖励 |
| **观测要点** | 这是最核心的训练进展指标 |

**训练阶段参考（以 GSM8K 为例）：**
| 阶段 | 典型 reward 范围 | 含义 |
|------|-----------------|------|
| 初始 | -0.3 ~ 0.0 | 模型答对率低，格式不规范 |
| 中期 | 0.0 ~ 0.3 | 开始学会正确格式和部分题目 |
| 后期 | 0.3 ~ 0.7 | 多数题目答对 |
| 收敛 | 0.6 ~ 0.9 | 接近最优 |

---

#### `reward_std` = 0.0533

| 属性 | 说明 |
|------|------|
| **含义** | 不同 prompt 组之间平均奖励的标准差 |
| **与 rewards/reward_fn/std 的区别** | 后者是组内标准差，reward_std 是**组间**标准差 |
| **观测要点** | 反映模型在不同难度题目上的表现差异 |

**诊断：**
- reward_std 持续减小 → 模型在所有题目上表现趋于一致（可能过拟合到特定模式）
- reward_std 增大 → 模型对部分题目变好，对部分变差（分化）

---

#### `frac_reward_zero_std` = 0.0

| 属性 | 说明 |
|------|------|
| **含义** | 组内奖励标准差为零的 prompt 比例 |
| **计算** | 即"所有 G 个回复得到完全相同奖励"的 prompt 占比 |
| **健康范围** | < 0.3 |
| **观测要点** | **极其重要的监控指标！** |

**深入解析：**

当某个 prompt 的 G 个回复全都得到相同的奖励时，GRPO 无法从中学到任何东西（优势全为 0）。

- `frac_reward_zero_std = 0.0` → **理想状态**，每组都有区分度
- `frac_reward_zero_std > 0.5` → **严重问题**，超过一半的训练数据产生不了梯度
- `frac_reward_zero_std = 1.0` → **训练失效**，所有组内回复完全相同

**常见原因：**
1. 奖励函数太粗糙（只有 0/1 两档），G 个回复全对或全错
2. Group size 太小，无法产生差异化回复
3. 温度太低，采样多样性不足
4. 模型已经收敛，对所有 prompt 给出相同答案

---

### 2.4 KL 散度与熵

#### `kl` = 0.000957

| 属性 | 说明 |
|------|------|
| **含义** | 当前策略 $\pi_\theta$ 与参考策略 $\pi_{\text{ref}}$ 之间的 **KL 散度** |
| **计算** | $D_{KL}(\pi_\theta \| \pi_{\text{ref}}) = \mathbb{E}_{\pi_\theta}\left[\log \frac{\pi_\theta(a|s)}{\pi_{\text{ref}}(a|s)}\right]$ |
| **健康范围** | 0.001 ~ 5.0（取决于 KL 系数 β 和训练阶段） |
| **观测要点** | 应缓慢增长，不应发散 |

**KL 散度分级标准：**

| KL 值范围 | 状态 | 说明 |
|-----------|------|------|
| < 0.01 | 极低 | 策略几乎没变化，可能学习率过低 |
| 0.01 ~ 1.0 | 健康 | 策略在稳步调整 |
| 1.0 ~ 5.0 | 偏高 | 策略已显著偏离，注意观察 |
| 5.0 ~ 10.0 | 高 | 接近危险区域，考虑增大 β |
| > 10.0 | **危险** | 策略严重偏离，可能导致退化 |

**本例中 KL = 0.000957：** 非常低，说明训练刚开始（epoch=0.02），策略还几乎没有变化，符合预期。

---

#### `entropy` = 1.2086

| 属性 | 说明 |
|------|------|
| **含义** | 当前策略的 **信息熵**，衡量输出分布的随机性/多样性 |
| **计算** | $H(\pi_\theta) = -\mathbb{E}_{\pi_\theta}[\log \pi_\theta(a|s)]$ |
| **健康范围** | 0.5 ~ 3.0（取决于词表大小和任务） |
| **观测要点** | 应缓慢下降但不应接近 0 |

**熵的意义：**
- **熵太高**（>5.0）→ 模型输出接近随机，还没学会
- **熵适中**（1.0~3.0）→ 模型有一定确定性但保持探索
- **熵太低**（<0.3）→ 模型过度确定，丧失多样性（策略崩塌前兆）

**entropy_coef（配置中 = 0.01）的作用：**
在 loss 中加入 $-\alpha \cdot H(\pi_\theta)$ 项，鼓励策略保持一定的探索性，防止过早收敛到单一模式。

---

### 2.5 Clip Ratio 相关

#### `clip_ratio/low_mean` = 0.0, `clip_ratio/low_min` = 0.0

| 属性 | 说明 |
|------|------|
| **含义** | 策略比率 $r(\theta) = \frac{\pi_\theta}{\pi_{\text{old}}}$ **低于下界** $(1-\epsilon)$ 被裁剪的比例 |
| **下界** | $1 - \epsilon = 1 - 0.2 = 0.8$ |
| **触发条件** | 当 $r(\theta) < 0.8$，即新策略大幅降低了某个 action 的概率 |

---

#### `clip_ratio/high_mean` = 0.0, `clip_ratio/high_max` = 0.0

| 属性 | 说明 |
|------|------|
| **含义** | 策略比率 $r(\theta)$ **超过上界** $(1+\epsilon)$ 被裁剪的比例 |
| **上界** | $1 + \epsilon = 1 + 0.2 = 1.2$ |
| **触发条件** | 当 $r(\theta) > 1.2$，即新策略大幅提高了某个 action 的概率 |

---

#### `clip_ratio/region_mean` = 0.0

| 属性 | 说明 |
|------|------|
| **含义** | 总的被裁剪比例（low + high） |
| **健康范围** | 0.0 ~ 0.3 |
| **观测要点** | 如果太高，说明策略更新过于激进 |

**本例中所有 clip_ratio = 0.0 的解释：**

训练刚开始（epoch=0.02），策略还几乎没有变化，$\pi_\theta \approx \pi_{\text{old}}$，所以 $r(\theta) \approx 1.0$，没有任何 action 触及裁剪边界。

**随着训练推进的典型变化：**

| 训练阶段 | clip_ratio 典型值 | 含义 |
|---------|------------------|------|
| 初期 | 0.0 ~ 0.01 | 策略变化极小 |
| 中期 | 0.05 ~ 0.15 | 正常学习中 |
| 后期 | 0.1 ~ 0.3 | 策略调整幅度较大 |
| 异常 | > 0.5 | 策略更新过于激进，需降低学习率 |

## 3. 训练信号监控

### 3.1 有效梯度信号检测

训练能否正常进行的**最基本条件**是：优化器能收到有意义的梯度。

| 检查项 | 判断标准 | 问题原因 |
|--------|---------|---------|
| loss 是否在下降 | 连续 50 步内应有明显下降趋势 | 学习率、奖励函数 |
| grad_norm 是否正常 | 0.1 < grad_norm < 50 | 梯度消失/爆炸 |
| frac_reward_zero_std | < 0.3 | 奖励函数区分度不足 |
| reward_std | > 0.01 | 组间差异不足 |

### 3.2 学习效率监控

```
学习效率 = Δreward / Δstep
```

| 阶段 | 期望学习效率 |
|------|------------|
| Warmup（前 3%） | 接近 0（正常） |
| 快速学习期 | 每 100 步 reward 增加 0.01~0.1 |
| 稳定期 | 每 100 步 reward 增加 < 0.01 |
| 过拟合期 | reward 不再上升或开始下降 |

### 3.3 GRPO 特有的信号质量

GRPO 的独特之处在于使用**组内相对排名**作为优势估计：

$$\hat{A}_i = \frac{r_i - \text{mean}(r_1, \dots, r_G)}{\text{std}(r_1, \dots, r_G) + \epsilon}$$

关键监控：
- **rewards/reward_fn/std > 0** → 组内有区分度，能产生有效优势信号
- **frac_reward_zero_std < 0.3** → 大部分组能产生梯度
- **completions 长度方差适中** → 模型在尝试不同策略

## 4. 稳定性监控

### 4.1 Loss 稳定性

| 现象 | 严重度 | 原因 | 处理方式 |
|------|--------|------|---------|
| Loss 缓慢下降 | ✅ 正常 | — | 继续训练 |
| Loss 轻微波动（±20%） | ✅ 正常 | RL 固有方差 | 无需处理 |
| Loss 大幅震荡（±100%） | ⚠️ 警告 | 学习率过大/批次太小 | 降低 lr 或增大 batch |
| Loss 突然飙升（>10x） | 🔴 严重 | 梯度爆炸 | 立即停训，降低 lr |
| Loss 变为 NaN | 🔴 严重 | 数值溢出 | 检查数据、降 lr、用 bf16 |

### 4.2 KL 散度稳定性

```python
# monitor.py 中的 KL 检查逻辑
if max_kl > 10.0:
    warnings.append("KL 散度过大，策略偏离参考模型过多")

if all(recent_kl[i] < recent_kl[i+1] for i in range(len(recent_kl)-1)):
    warnings.append("KL 散度持续增长，策略可能在 exploitation")
```

**KL 稳定性分级：**
| 行为 | 评估 | 措施 |
|------|------|------|
| KL 缓慢线性增长 | 正常 | 继续 |
| KL 指数级增长 | 危险 | 增大 β（KL 系数） |
| KL 突然跳变 | 异常 | 检查该步骤的数据 |
| KL 始终接近 0 | 低效 | 增大 lr 或减小 β |

### 4.3 梯度稳定性

**梯度范数的滑动窗口监控：**
```python
# 建议的监控逻辑
window = grad_norms[-50:]
mean_grad = np.mean(window)
std_grad = np.std(window)

if std_grad / mean_grad > 2.0:
    print("⚠️ 梯度不稳定：变异系数过大")
if max(window) / min(window) > 100:
    print("⚠️ 梯度范围过大，训练不稳定")
```

### 6.1 奖励趋势分析

```python
# 来自 monitor.py 的奖励欺骗检测
first_half = rewards[:len(rewards)//2]
second_half = rewards[len(rewards)//2:]

first_slope = (first_half[-1] - first_half[0]) / len(first_half)
```

### 7.1 训练集 vs 测试集表现

| 情况 | 判断 |
|------|------|
| 训练 reward ↑，测试 reward ↑ | 正常学习 |

### 7.2 分布外（OOD）鲁棒性

GRPO 训练可能导致的泛化风险：
1. **模式固化**：模型学会了特定的回答模板，换个问法就不会了
2. **奖励拟合**：模型找到了 hack 奖励函数的捷径
3. **长度偏好**：模型学会了生成特定长度以获取格式奖励

## 8. 风险与异常监控

### 8.1 奖励欺骗（Reward Hacking）

**这是 RL 训练中最重要也最隐蔽的风险。**

来自 `monitor.py` 的三重检验：

| 检验 | 条件 | 说明 |
|------|------|------|
| 加速上升检验 | 后半段斜率 > 前半段 × 2 | 奖励异常加速 |
| KL 发散检验 | max KL > 10 或持续递增 | 策略过度偏离 |
| 方差坍缩检验 | 后期方差 < 前期 × 0.1 | 收敛到固定模式 |

**处理方式：**
```
1. 增大 KL 系数 β（从 0.05 → 0.1 或更大）
2. 使用更鲁棒的奖励函数
3. 添加多样性正则化（增大 entropy_coef）
4. 人工检查生成样本
5. 早停（使用测试集表现作为停止准则）
```

### 8.2 策略崩塌（Policy Collapse）

| 信号 | 表现 |
|------|------|
| entropy → 0 | 模型对所有 prompt 输出几乎相同的内容 |
| frac_reward_zero_std → 1.0 | 所有组内回复完全一样 |
| completions 长度方差 → 0 | 所有生成长度相同 |
| KL 激增 | 策略剧烈偏离参考模型 |

### 8.3 梯度异常

| 类型 | 表现 | 原因 | 处理 |
|------|------|------|------|
| 梯度爆炸 | grad_norm > 100, loss = NaN | lr 过大/数据异常 | 降 lr, 增强 clip |
| 梯度消失 | grad_norm < 0.001 | 奖励信号太弱 | 调整奖励函数 |
| 梯度震荡 | grad_norm 方差极大 | batch 太小 | 增大 gradient_accumulation |

### 8.4 数值不稳定

| 问题 | 预防措施 |
|------|---------|
| loss = NaN | 使用 bf16, 梯度裁剪, 检查数据 |
| KL = inf | 限制策略比率, 增大 epsilon |
| reward 溢出 | 奖励函数加 clamp |

## 10. 实战经验与调参指南

### 10.1 常见问题快速诊断

| 现象 | 最可能原因 | 第一步处理 |
|------|-----------|-----------|
| reward 不上升 | 学习率过低 / 奖励信号无区分度 | 增大 lr 或调整奖励函数 |
| reward 上升但准确率不变 | 奖励欺骗 | 增大 β，检查 reward 设计 |
| loss = NaN | 梯度爆炸 | 降低 lr，增强 grad_clip |
| entropy 快速下降 | 策略崩塌 | 增大 entropy_coef |
| KL 持续飙升 | β 太小 | 增大 β（如 0.05→0.1） |
| frac_reward_zero_std 高 | group_size 太小 / 温度太低 | 增大 G 或 temperature |
| clipped_ratio 高 | 生成被截断 | 增大 max_completion_length |
| 训练太慢 | batch 太小 / 模型太大 | 增大 gradient_accumulation |

### 10.2 超参数调整优先级

```
高优先级（影响训练成败）：
  1. learning_rate: 5e-7 → 范围 [1e-7, 5e-6]
  2. kl_coef (β): 0.05 → 范围 [0.01, 0.5]
  3. group_size (G): 8 → 范围 [4, 32]

中优先级（影响训练效率）：
  4. per_device_train_batch_size × gradient_accumulation_steps
  5. max_completion_length: 512
  6. temperature: 0.7 → 范围 [0.5, 1.0]

低优先级（微调）：
  7. clip_ratio (ε): 0.2 → 范围 [0.1, 0.3]
  8. entropy_coef: 0.01 → 范围 [0.001, 0.1]
  9. warmup_ratio: 0.03
```

### 10.3 不同 GPU 显存下的配置建议

| GPU | 推荐 batch | 推荐 group_size | gradient_accumulation | 预计显存 |
|-----|-----------|----------------|----------------------|---------|
| 24G (4090) | 1 | 4 | 16 | ~22GB |
| 48G (A6000) | 4 | 8-16 | 4 | ~40GB |
| 80G (A100) | 8 | 16-32 | 2 | ~70GB |

### 10.4 训练阶段经验参考

| 阶段 | 步数（5000样本）| 特征 | 注意事项 |
|------|---------------|------|---------|
| Warmup | 0 ~ 30 步 | loss 不降, reward 不升 | 正常，不要急 |
| 快速学习 | 30 ~ 300 步 | reward 快速上升 | 黄金时期，密切监控 |
| 稳定调整 | 300 ~ 800 步 | reward 缓慢上升 | 注意过拟合信号 |
| 收敛 | 800+ 步 | reward 基本不变 | 可以考虑停训 |

### 10.5 关键经验总结

1. **RL 训练不稳定是常态**：Loss 波动比 SFT 大得多，不要过度紧张
2. **KL 散度是安全阀**：当 KL 开始发散，优先增大 β
3. **奖励设计比算法更重要**：好的奖励函数能让简单配置也训好
4. **Group Size 越大越稳定**：但显存消耗也线性增长
5. **早停比多训重要**：过了收敛点继续训练大概率是负收益
6. **定期抽查生成质量**：数字指标可能骗人，实际生成不会

```python
def monitor_token_probs(logits, labels):
    """监控模型对正确 token 的置信度变化"""
    probs = F.softmax(logits, dim=-1)
    correct_probs = probs.gather(-1, labels.unsqueeze(-1))
    return {
        "mean_correct_prob": correct_probs.mean().item(),
        "min_correct_prob": correct_probs.min().item(),

# 2. 奖励分布直方图
def reward_distribution(rewards):
    """分析奖励分数的分布形状"""
    import numpy as np
    return {
        "p25": np.percentile(rewards, 25),
        "p50": np.percentile(rewards, 50),
        "p75": np.percentile(rewards, 75),
        "skewness": scipy.stats.skew(rewards),
    }

# 3. 生成重复度检测
def repetition_score(text):
    """检测生成文本中的重复"""
    ngrams = [text[i:i+4] for i in range(len(text)-3)]
    unique_ratio = len(set(ngrams)) / len(ngrams) if ngrams else 1.0
    return 1.0 - unique_ratio  # 越高越重复
```
